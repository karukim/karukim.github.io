# How does a designer start a GitHub blog?
디자이너의 깃허브 블로그 입성기: 환경 구축과 기본 개념의 이해


## 맥북의 3단 레이어 구조
디자인 작업 시 '레이어'를 쌓는 것과 똑같습니다.

1. Hardware (하드웨어): Mac
가장 밑바닥에 있는 물리적인 실체입니다. 튼튼한 캔버스와 같죠.

손으로 만질 수 있는 기계 그 자체입니다.
예시: 맥북의 알루미늄 바디, M3 칩(CPU), 레티나 디스플레이, 키보드, SSD 저장 장치.
역할: 소프트웨어가 실행될 수 있는 물리적인 공간과 연산 능력을 제공합니다.


2. OS (운영체제): macOS
하드웨어라는 캔버스 위에 깔리는 기본 배경입니다. 하드웨어를 제어하고 우리가 프로그램을 돌릴 수 있게 판을 깔아주는 역할을 합니다.

하드웨어 바로 위에서 하드웨어를 관리하고 사용자 인터페이스를 제공하는 소프트웨어입니다.
정확한 명칭: 맥(Mac) 기기에서 사용하는 운영체제는 macOS입니다. (iOS는 아이폰 전용입니다.)
역할: 하드웨어(화면, 메모리, 프로세서)를 직접 제어하며, 우리가 사용하는 일반 프로그램들이 안정적으로 돌아가도록 '관리자' 역할을 수행합니다.


3. Program/App (프로그램): Chrome, VS Code, Jekyll
운영체제라는 판 위에서 실제로 우리가 사용하는 도구들입니다. 크롬 브라우저로 정보를 찾고, VS Code로 글을 쓰는 식입니다.

우리가 특정 목적을 위해 실행하는 개별 프로그램들입니다.
예시: Chrome(웹 서핑), VS Code(코드 편집), Photoshop(디자인), Slack(협업).
역할: 운영체제 위에서 작동하며 사용자가 원하는 구체적인 작업(웹 서핑, 블로그 수정 등)을 수행합니다.



## 데이터의 여정: 로컬에서 서버로, 다시 세상으로
내가 쓴 글이 어떻게 전 세계(그리고 내 브라우저)에 보여지는지, 그 층위별 이동 경로입니다.

1. 데이터 배포 (Deployment)
로컬 환경(내 Mac)에서 작성된 마크다운(.md) 파일과 설정값들을 git push 명령어를 통해 GitHub 원격 저장소로 **전송(Transmit)**합니다. 이 과정에서 데이터는 네트워크를 통해 서버로 **업로드(Upload)**됩니다.

2. 정적 사이트 생성 및 호스팅 (Build & Hosting)
GitHub 서버는 수신된 데이터를 바탕으로 Jekyll 엔진을 가동하여 HTML 파일을 생성하는 빌드(Build) 과정을 거칩니다. 이후 완성된 웹사이트를 인터넷상에 게시하는 **배포(Deployment)**를 완료합니다.

3. 데이터 요청과 응답 (Request & Response)
사용자(나 혹은 방문자)가 웹 브라우저 주소창에 블로그 URL을 입력하면, 브라우저는 GitHub 서버에 해당 페이지의 데이터를 보내달라는 **요청(Request)**을 보냅니다.

![alt text](image.png)

4. 데이터 서빙 및 렌더링 (Serving & Rendering)
서버는 요청을 확인하고 저장된 HTML/CSS 데이터를 브라우저로 응답(Response) 혹은 전송합니다. 최종적으로 브라우저는 수신된 데이터를 해석하여 화면에 시각적으로 구현하는 렌더링(Rendering) 과정을 수행합니다.



## 주소의 차이가 만드는 공개의 범위: localhost와 github.io
로컬 환경과 배포 환경의 이해

**Local vs Production: 내 맥(Mac)과 깃허브 서버의 경계**
지킬(Jekyll) 블로그 운영 시 데이터가 머무는 환경은 크게 두 가지로 분리됩니다.
이 둘의 차이를 이해하는 것은 안정적인 배포 프로세스를 구축하는 핵심입니다.

1. 로컬 환경 (Local Environment)
정의: 사용자의 물리적 하드웨어(Mac) 내부에서만 구동되는 독립적인 작업 환경입니다.

접근 주소: [http://127.0.0.1:4000](http://127.0.0.1:4000) (또는 localhost:4000)

기술적 특징: 내 컴퓨터의 자원(CPU, RAM)을 사용하여 임시 웹 서버를 실행합니다. 외부 네트워크와 단절되어 있어 실시간 수정 및 에러 디버깅이 자유로우며, git push를 하기 전까지는 외부로 데이터가 유출되지 않습니다.

2. 배포 환경 (Production Environment)
정의: 전 세계 사용자가 접속할 수 있도록 GitHub의 원격 서버(Remote Server)에 최적화되어 올라간 완성본 환경입니다.

접근 주소: [https://karukim.github.io](https://karukim.github.io)

기술적 특징: GitHub Pages 서비스가 호스팅을 담당합니다. 사용자가 git push를 통해 데이터를 전송하면, 서버에서 빌드(Build) 과정을 거쳐 정적 파일로 변환된 후 전 세계로 **서빙(Serving)**됩니다.


## 데이터 전송 프로세스
파일이 실제 웹사이트에 반영되기까지의 단계입니다.

1. 스테이징 영역(Staging Area)의 정의
스크린샷에서 파일 옆의 + 아이콘을 클릭하는 행위는 해당 파일을 Staged Changes 상태로 만드는 것입니다.

정확한 명칭: 스테이징(Staging) 또는 **인덱싱(Indexing)**이라 부릅니다.

기술적 의미: 작업 디렉토리(내 Mac)에서 수정된 파일 중, 다음 커밋(Commit)에 포함할 파일들만 선별하여 준비 영역에 올리는 동작입니다.

장바구니 비유: 쇼핑몰에서 물건을 고른 뒤 결제하기 전 '장바구니'에 담아두는 것과 같습니다. 장바구니에 담았다고 해서 결제(커밋)가 완료된 것은 아니며, 마음이 바뀌면 다시 뺄 수도 있습니다.

2. 커밋(Commit)과 푸시(Push)의 차이
Commit하면 바로 업로드될까?

- Commit (커밋): 스테이징 영역에 담긴 파일들을 하나의 '버전'으로 묶어 **로컬 저장소(내 Mac)**에 확정 기록하는 것입니다. 이 단계에서는 아직 인터넷(GitHub)으로 전송되지 않습니다.

- Push (푸시): 내 컴퓨터에 기록된 커밋들을 **GitHub 원격 저장소(Remote Repository)**로 실제 전송하는 단계입니다. 이 과정이 끝나야 비로소 파일이 업로드됩니다.

- Graph 영역의 의미
GRAPH 섹션은 지금까지 발생한 커밋(Commit)들의 역사를 시각적으로 보여주는 타임라인입니다.
각 점과 선은 '누가, 언제, 어떤 내용을 확정(Commit)했는지'를 나타냅니다.
사용자께서 +를 눌러 스테이징한다고 해서 그래프에 즉시 나타나는 것이 아니라,
Commit 버튼을 눌러 한 버전이 완성되었을 때 그래프에 새로운 점이 추가됩니다.

- 요약
- + 클릭 (Stage): 결제 전 장바구니 담기.
- Commit: 영수증 발행 및 내 장부에 기록 (로컬 저장소 저장).
- Push / Sync: 물건을 배송지로 발송 (GitHub 서버 전송 및 배포 환경 동기화).
